## TL;DR

**Phase 4 (per-node marking customisation) now works.** Custom Bezier-arc markings
are rendered between user-selected endpoint pairs on intersection nodes via
`Graphics.DrawMesh` from a managed `MarkingMeshRenderSystem`. The breakthrough
was discovering that HDRP requires **`HDMaterial.ValidateMaterial(material)`** on
every freshly constructed Material вҖ” without it all shader passes silently
no-op. Vanilla CS2 calls this in `OverlayRenderSystem`, `ManagedBatchSystem`,
`AssetImportPipeline`; no public Unity docs mention it.

## Current state (commit `385b545`)

### What works end-to-end

1. **Hotkey Ctrl+M** activates `MarkingNodeToolSystem`.
2. **Click a node** вҶ’ `MarkingEndpointExtractor` returns N+1 endpoints per edge
   (gap-based, between lane stitches + outer kerbs).
3. **Click endpoint вҶ’ click endpoint** вҶ’ toggles a `MarkingPair` on the node's
   buffer (Traffic-style same-gesture create+remove).
4. **`MarkingMeshRenderSystem`** (Modification at `SystemUpdatePhase.Rendering`)
   diffs the `MarkingPair` buffers against a `Dictionary<PairKey, Mesh>` cache.
   Each pair becomes a flat quad (PoC), built procedurally with explicit
   y-expanded bounds.
5. Per-frame `RenderPipelineManager.beginContextRendering` callback dispatches
   `Graphics.DrawMesh` for every cache entry Г— every Game/SceneView camera.
6. Material: `new Material(Shader.Find("HDRP/Unlit"))` + `_SurfaceType=Opaque`,
   `_DoubleSidedEnable=1`, `_CullMode=Off`, then **`HDMaterial.ValidateMaterial`**.

### What's still PoC-quality

- **Geometry**: flat 1m Г— Nm quad вҖ” not a Bezier ribbon yet. Sits 0.5m above
  road for visibility.
- **Material**: plain white `HDRP/Unlit` opaque, not the realistic road-decal
  look. No alpha blend with road surface.
- **Styles**: only one вҖ” single solid line. IMT has ~20.
- **Persistence**: `MarkingPair` buffer is `IBufferElementData` with `ISerializable`
  v2 вҖ” *should* survive save/reload, **not verified yet**.

## The breakthrough вҖ” `HDMaterial.ValidateMaterial`

### Why it was hidden so well

- Not in Unity public docs.
- Not in HDRP package samples.
- Required only because we construct Materials at runtime вҖ” every Material asset
  shipped by the game was validated by the Unity Editor at build time.
- No errors, no warnings, no shader-compilation log вҖ” just silent zero pixels.

### The 60-second recipe

```csharp
using UnityEngine.Rendering.HighDefinition;

var mat = new Material(Shader.Find("HDRP/Unlit"));
mat.color = Color.white;
mat.SetFloat("_SurfaceType", 0f);          // 0 = Opaque, 1 = Transparent
mat.SetFloat("_DoubleSidedEnable", 1f);
mat.SetFloat("_CullMode", 0f);             // Off
HDMaterial.ValidateMaterial(mat);          // вҶҗ the missing call
```

The `ValidateMaterial` call derives shader keywords (`_SURFACE_TYPE_OPAQUE`,
`_BLENDMODE_*`, `_DISABLE_DECALS`, `_DOUBLESIDED_ON`, вҖҰ), `renderQueue`, and
stencil state from the property values above. Without it, no keyword is
enabled, every render-pass variant is "empty" code, DrawMesh dispatches but
produces zero output.

### Bonus gotcha вҖ” flat quad bounds

HDRP Forward+ frustum culling drops meshes whose AABB collapses on any axis.
Our 4-vertex quad has `bounds.size.y == 0` (all verts share Y). Workaround
after `RecalculateBounds()`:

```csharp
var b = mesh.bounds;
b.Expand(new Vector3(0f, 1f, 0f));
mesh.bounds = b;
```

Vanilla side-steps this by passing a `Matrix4x4.TRS(..., scale)` with non-unit Y
to DrawMesh; we use identity matrix, so we need explicit bounds expansion.

### Why community never found it

Two reasons combined:
- CS2 modding community is small; nobody had `IMT MacSergey-tier` motivation
  to fight through HDRP runtime material setup
- Vanilla CS2 uses `ValidateMaterial` deep inside infrastructure systems
  (`OverlayRenderSystem.cs:804`, `ManagedBatchSystem.cs:1392`) вҖ” not in the
  obvious "rendering example" path

We found it by chaining 3 research agents and asking the right narrow question:
"what *exactly* does vanilla do between `new Material(...)` and visible pixels?"

## Roadmap toward IMT-equivalent feature set

Goal stated by user: turn the chaos of vanilla markings on complex intersections
into manageable, customizable, beautiful markings. The "kill-feature" example:
4-lane avenue with median, crossed by 2-lane road, with a left-turn from far
side вҖ” the user needs to delete the median *segment* that crosses the turn lane.

That's exactly what IMT does in CS1 via line-intersection segmentation.

### Step-by-step plan (incremental, each step independently shippable)

1. **Bezier ribbon mesh** (today, ~1h): replace flat quad with N-segment ribbon
   along Bezier curve. Reuses `chord*0.4` pull from `MarkingOverlaySystem`.
   Visual: smooth arcs through intersections.

2. **Realistic look** (~2h): swap `HDRP/Unlit` for road-decal material with
   alpha; tune height to ~0.02m (decal-like, not floating bar). Will need to
   experiment with which vanilla material can be borrowed *after* ValidateMaterial.

3. **Line styles** (~1 day): solid, dashed, double-solid, stop line. Each style
   = one mesh generation function. Style stored per pair in buffer extension.

4. **Default markings generator** (~1 day): button "generate vanilla-equivalent
   markings from NetCompositionLane" so user doesn't have to draw every line
   manually. Output goes into the same MarkingPair buffer вҖ” fully editable.

5. **Line-line intersection math** (~1 day): for every pair of lines on a node,
   compute Bezier-Bezier intersection points (LUT-based, ~16 samples per line
   is enough). Store as `(lineA_pair_key, t_a, lineB_pair_key, t_b)` tuples.

6. **Segment delete UI** (~half day): click on a segment between two
   intersections вҶ’ remove `(line_key, t_start, t_end)` range from cache.
   Render skips deleted ranges by emitting two separate sub-meshes per line.

7. **Filler areas, decals, symbols** (IMT-tier): per user demand. Not
   blocking the core killer feature.

### Time estimate

- Steps 1вҖ“3: cosmetic polish, ~half day total
- Steps 4вҖ“6: the actual killer feature (segment-aware markings), ~3 days
- Step 7: open-ended, as user wants

The fundamental tech wall is now behind us. Everything ahead is regular
incremental programming, no more research stalls.

## Open questions / known unknowns

1. **Save/reload persistence**: `MarkingPair` is `IBufferElementData` with
   `ISerializable` (schema v2). Should work, **not tested**. First user
   save/reload after PoC is the test.

2. **Material visual fidelity**: `HDRP/Unlit` looks like an opaque flat
   strip вҖ” not like a road decal. Need experimentation with what material/shader
   combination gives the "painted on road" look while still surviving
   `ValidateMaterial`. Some vanilla materials may suffice when we copy + revalidate.

3. **Performance at scale**: PoC renders one DrawMesh per pair per camera per
   frame. Untested with hundreds of pairs across a large city. If it becomes a
   bottleneck, switch to `Graphics.DrawMeshInstancedIndirect` (vanilla uses
   this pattern in `RouteRenderSystem.cs:262`, `OverlayRenderSystem.cs:903-913`).

4. **Layer 4 reapply crash** (older issue from Phases 1/2): ~10-20s after style
   change game crashes. Documented earlier as K5 risk in
   `IMPLEMENTATION_PLAN.md`, marked deferred. Independent of mesh-render work.

## What I would tell my future self after `/compact`

- The mod **works** at PoC level. Don't think it's broken вҖ” first run with the
  user's save should show colored bars between selected endpoints.
- The single non-obvious piece is `HDMaterial.ValidateMaterial`. Memorize that
  function name. Without it, no HDRP material rendered from mod code ever
  renders anything.
- `Resources.FindObjectsOfTypeAll<Material>()` reveals the runtime shader
  landscape (helpful for picking what to deep-copy). In CS2 the histogram is
  dominated by `BH/SG_DefaultShader`, `BH/Decals/*`, `Shader Graphs/*`,
  `Hidden/VFX/*` вҖ” most coupled to vanilla render systems. Plain `HDRP/Unlit`
  was only found once (`AreaLightCookieViewer`). For our use, building fresh
  via `Shader.Find` + Validate is cleaner than deep-copying.
- `_refs/IMT/` has the CS1 NodeMarkup source vendored (~99 files). When in
  doubt about "how would a marking tool solve X", grep there first вҖ” but
  remember CS1 was built-in pipeline, not HDRP, so material recipes don't
  transfer; only architecture does.
- Implementation plan is at `IMPLEMENTATION_PLAN.md` (~585 lines). It has a new
  "Custom-mesh pivot" section at the top reflecting today's architectural shift;
  the historical 4-layer plan below is preserved for context.
